## Creating the ReplicaSet Definition File

Create a new directory called `replica-sets` and inside it create a file called `replicaset.yaml`.

### `replicaset.yaml`

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: myapp-replicaset
  labels:
    app: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      env: production
  template:
    metadata:
      name: myapp-pod
      labels:
        env: production
    spec:
      containers:
        - name: nginx-container
          image: nginx
```

**How this file was built step by step:**

- `apiVersion` → `apps/v1` (required for ReplicaSet)
- `kind` → `ReplicaSet`
- `metadata.name` → `myapp-replicaset`
- `metadata.labels` → `app: myapp`
- `spec.replicas` → `3`
- `spec.selector.matchLabels` → copied from the pod definition file (`env: production`)
- `spec.template` → copied the `metadata` and `spec` sections from the existing pod definition file and fixed indentation

> **Indentation fix tip:** After pasting the pod contents under `template`, select the pasted section and press **Tab twice** to fix indentation. Some editors do this automatically — there is also an extension called **Paste and Indent** that can help.

---

### Important — Label Matching Rule

There are two places where labels appear in the ReplicaSet file:

1. **`spec.selector.matchLabels`** — the selector the ReplicaSet uses to identify which pods to manage
2. **`spec.template.metadata.labels`** — the labels assigned to the pods created by the ReplicaSet

**These two must always be the same.** This is what ties the pods to the ReplicaSet.

> The label on the ReplicaSet itself (under `metadata.labels`) does not matter for this. Only the pod labels and the selector labels must match.

## Applying the ReplicaSet

**Verify the file contents:**
```bash
cat replica-sets/replicaset.yaml
```

**Create the ReplicaSet:**
```bash
kubectl create -f replica-sets/replicaset.yaml
```

**Check the ReplicaSet status:**
```bash
kubectl get replicaset
```

Output shows:
- **DESIRED** — number of replicas specified (3)
- **CURRENT** — number currently running (3)
- **READY** — number ready (3)

**Check the pods:**
```bash
kubectl get pods
```

You will see **3 pods** running. All pod names begin with `myapp-replicaset-` — this tells you they belong to the ReplicaSet and are not standalone pods.

**Inspect the ReplicaSet in detail:**
```bash
kubectl describe replicaset myapp-replicaset
```

This shows:
- Name of the ReplicaSet
- Selector labels
- Labels on the ReplicaSet itself
- Pod labels and container definition
- **Events** — full history of what happened (pods created, deleted, replaced, etc.)

## Verifying Self-Healing

To see the ReplicaSet's self-healing in action, delete one of the running pods:

```bash
kubectl delete pod myapp-replicaset-8nxxl
```

Wait a few seconds for the pod to fully terminate, then check:

```bash
kubectl get pods
```

You will notice:
- The deleted pod (`8nxxl`) is gone
- A **new pod with a different name** has been created automatically
- The new pod shows a recent creation time (e.g., "15 seconds ago")
- Total pod count is still **3**

The ReplicaSet detected that the number of available pods dropped below the desired count and immediately spun up a replacement. This is visible in the events when you run `kubectl describe replicaset myapp-replicaset`.

## What Happens When You Create a Pod with the Same Label?

If you manually create a pod that has the **same labels** as the ReplicaSet selector, the ReplicaSet will **terminate it immediately** — because it already has the desired number of replicas running and does not allow more.

**Example:** Create a pod called `nginx-2` with the label `env: production`:

```bash
kubectl create -f pods/nginx.yaml
```

Then check pods:

```bash
kubectl get pods
```

You will see `nginx-2` briefly in a **Terminating** state. The ReplicaSet controller deletes it right away.

Running `kubectl describe replicaset myapp-replicaset` will show this in the **Events** section — the ReplicaSet controller deleted the `nginx-2` pod that was just created.

> The ReplicaSet does not allow more pods with matching labels than the configured replica count.

## Updating a ReplicaSet — Scaling

### Method 1 — `kubectl edit`

```bash
kubectl edit replicaset myapp-replicaset
```

This opens the **live running configuration** of the ReplicaSet in a text editor (Vim by default).

> **Note:** This is **not** the original file you created. It is a temporary file generated by Kubernetes in memory. It contains many additional fields beyond what you originally provided. Changes saved here are **applied immediately** to the cluster, so be careful.

Scroll down to the `spec` section, find `replicas`, change the value (e.g., from `3` to `4`), then save and exit:

```
Esc → :wq → Enter
```

After saving, a new pod is spun up automatically to match the new replica count:

```bash
kubectl get pods
```

The same approach works for **scaling down** as well.

---

### Method 2 — `kubectl scale`

You can also scale without editing the file directly:

```bash
kubectl scale replicaset myapp-replicaset --replicas=2
```

> **Note:** Take care of the **double dashes** (`--`) before `replicas`.

You can specify a number greater or less than the current replica count.

Check the result:

```bash
kubectl get pods
```

The ReplicaSet will terminate pods to scale down (or create new ones to scale up) to match the number you specified.
