## What is a Controller?

Controllers are the **brain behind Kubernetes**. They are processes that monitor Kubernetes objects and respond accordingly.

---

## Why Do We Need a Replication Controller?

### High Availability

Consider a scenario where we have a **single pod** running our application. If the application crashes and the pod fails, users will no longer be able to access the application.

To prevent this, we want **more than one instance or pod running at the same time**. That way, if one fails, the application is still running on the other one.

The replication controller helps us run **multiple instances of a single pod** in the Kubernetes cluster, thus providing **high availability**.

> **Does that mean you can't use a replication controller if you plan to have a single pod?**
> No. Even if you have a single pod, the replication controller can help by **automatically bringing up a new pod** when the existing one fails.

The replication controller ensures that the **specified number of pods are running at all times** — even if it's just 1 or 100.

---

### Load Balancing and Scaling

Another reason we need a replication controller is to **create multiple pods to share the load** across them.

For example:

- We have a **single pod** serving a set of users.
- When the number of users increases, we **deploy additional pods** to balance the load across them.
- If the demand increases further and we run out of resources on the first node, we can **deploy additional pods across other nodes** in the cluster.

The replication controller **spans across multiple nodes** in the cluster. It helps us:

- Balance the load across multiple pods on different nodes
- Scale our application when demand increases

---

## Replication Controller vs ReplicaSet

Both have the same purpose, but they are **not the same**:

| | Replication Controller | ReplicaSet |
|---|---|---|
| Status | Older technology — being replaced | New recommended way |
| Use going forward | Avoid | Preferred |

Whatever applies to the Replication Controller remains applicable to ReplicaSet as well. There are minor differences in the way each works. Going forward, we will **stick to ReplicaSets** in all demos and implementations.

## Creating a Replication Controller

We start by creating a replication controller definition file named **`rc-definition.yaml`**.

### The Four Sections

Like any Kubernetes definition file, it has four sections:

| Field | Value |
|-------|-------|
| `apiVersion` | `v1` — Replication Controller is supported in Kubernetes API version v1 |
| `kind` | `ReplicationController` |
| `metadata.name` | `myapp-rc` |
| `metadata.labels` | `app` and `type` with assigned values |

---

### The Template Section

The **spec section** is the most crucial part of the definition file. It defines what's inside the object we are creating.

Since the replication controller creates multiple instances of a pod, we need to tell it — **what pod?**

We create a **`template`** section under `spec` to provide a pod template for the replication controller to use when creating replicas.

To define the pod template, we reuse the contents of our existing pod definition file — **everything except `apiVersion` and `kind`** — and place it under the `template` section.

> Everything moved into `template` must be **indented to the right** and have more spaces before them than the `template` line itself. They must be **children of the template section**.

This results in two `metadata` sections and two `spec` sections in the file — one for the replication controller and one for the pod. We have essentially **nested two definition files together**, with the replication controller as the parent and the pod definition as the child.

---

### The Replicas Field

We also need to specify **how many replicas** we need. Add a `replicas` property under `spec`.

> **Note:** `template` and `replicas` are **direct children of `spec`** — they are siblings and must be on the same vertical line, meaning they must have an equal number of spaces before them.

---

## Complete YAML — `rc-definition.yaml`

```yaml
apiVersion: v1
kind: ReplicationController
metadata:
  name: myapp-rc
  labels:
    app: myapp
    type: front-end
spec:
  replicas: 3
  template:
    metadata:
      name: myapp-pod
      labels:
        app: myapp
        type: front-end
    spec:
      containers:
        - name: nginx-container
          image: nginx
```

**Create the Replication Controller:**
```bash
kubectl create -f rc-definition.yaml
```

When created, the replication controller immediately creates pods using the pod definition template — as many as required (3 in this case).

**View the list of Replication Controllers:**
```bash
kubectl get replicationcontroller
```

The output shows the desired number of replicas, the current number of replicas, and how many of them are ready.

**View pods created by the Replication Controller:**
```bash
kubectl get pods
```

You will see three pods running, all starting with the name of the replication controller — **`myapp-rc`** — indicating they were all created automatically by it.

## Creating a ReplicaSet

### Complete YAML — `replicaset-definition.yaml`

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: myapp-replicaset
  labels:
    app: myapp
    type: front-end
spec:
  template:
    metadata:
      name: myapp-pod
      labels:
        app: myapp
        type: front-end
    spec:
      containers:
        - name: nginx-container
          image: nginx
  replicas: 6
  selector:
    matchLabels:
      type: front-end
```

---

## Commands

```bash
# Create a ReplicaSet
kubectl create -f replicaset-definition.yml

# View ReplicaSets
kubectl get replicaset

# Delete a ReplicaSet (also deletes all underlying PODs)
kubectl delete replicaset myapp-replicaset

# Replace / Update a ReplicaSet
kubectl replace -f replicaset-definition.yml

# Scale using the file
kubectl scale --replicas=6 -f replicaset-definition.yml
```

---

## Scaling a ReplicaSet

There are two ways to scale a ReplicaSet:

**Method 1 — Using the file:**
```bash
kubectl scale --replicas=6 -f replicaset-definition.yml
```

**Method 2 — Using TYPE and NAME:**
```bash
kubectl scale --replicas=6 replicaset myapp-replicaset
```

> In Method 2, `replicaset` is the **TYPE** and `myapp-replicaset` is the **NAME**.
